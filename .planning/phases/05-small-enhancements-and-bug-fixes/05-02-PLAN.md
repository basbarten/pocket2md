Update output stream usage throughout cli.js:

**Stderr (console.error) for:**
- All error messages in fetchArticleContent catch blocks
- All warning messages (console.warn → console.error)
- Malformed YAML warnings
- File operation errors
- Unexpected errors

**Stdout (console.log) for:**
- Progress messages ("Processing article N/M...")
- Success messages ("Fetching: URL", "✅ Successfully processed...")
- Summary statistics
- Startup messages

Enhance error categorization in fetchArticleContent():

Current code has basic categorization but shows generic messages. Enhance to show specific error types:

**Network errors (catch block):**
- `error.name === 'AbortError'` → "Timeout (30s)"
- `error.code === 'ENOTFOUND'` → "DNS resolution failed"
- `error.code === 'ECONNREFUSED'` → "Connection refused"
- `error.code === 'ECONNRESET'` → "Connection reset"
- `error.code === 'ETIMEDOUT'` → "Connection timeout"
- `error.message.includes('certificate')` → "SSL certificate error"
- `error.message.includes('redirect')` → "Too many redirects"
- Otherwise → Use error.message or "Unknown network error"

**HTTP errors (response.ok check):**
- `status === 404` → "Content not found (404)"
- `status === 429` → "Rate limited (429)"
- `status === 500` → "Server error (500)"
- `status === 503` → "Service unavailable (503)"
- `status >= 400 && < 500` → `Client error (${status})`
- `status >= 500` → `Server error (${status})`

**Error message format:**
`console.error(\`Failed to fetch "\${title}" (\${url}) - \${specificErrorType}\`);`

Keep return null behavior - don't change error handling flow, just improve error messages.

Add error collection and log file writing to processArticles():

1. **Create error collection array at start of processArticles():**
   ```javascript
   const failedArticles = [];
   ```

2. **Collect failed articles in processing loop:**
   
   When fetchArticleContent returns null (error occurred):
   ```javascript
   failedArticles.push({
     index: i + 1,
     title: row.title || 'Untitled',
     url: row.url,
     error: lastErrorMessage, // Capture from fetchArticleContent
     timestamp: new Date().toISOString()
   });
   ```
   
   Modify fetchArticleContent to return error detail (not just null):
   - Change return type to return `{content: string, error: string}` or `{content: null, error: string}`
   - Update all callers to check `result.content` instead of direct null check

3. **Write error log file after processing completes:**
   
   After summary output, before success message:
   ```javascript
   if (failedArticles.length > 0) {
     const timestamp = new Date().toISOString().replace(/[:.]/g, '-').substring(0, 19);
     const errorLogPath = path.join(parsedArgs.output, `errors-${timestamp}.log`);
     
     const logContent = [
       `Pocket2md Error Log - ${new Date().toISOString()}`,
       `Total errors: ${failedArticles.length}`,
       '',
       ...failedArticles.map(article => 
         `[${article.index}] "${article.title}" (${article.url}) - ${article.error}`
       )
     ].join('\n');
     
     fs.writeFileSync(errorLogPath, logContent, 'utf8');
     console.error(`Error log written to: ${errorLogPath}`);
   }
   ```

4. **Error log file format:**
   - Location: `./output/errors-YYYY-MM-DDTHH-MM-SS.log`
   - Plain text, one error per line
   - Include: article index, title, URL, error type
   - Header with timestamp and total count
   - Only created if errors occurred (don't create empty logs)

Preserve all existing error handling behavior - this just adds audit trail, doesn't change flow.
